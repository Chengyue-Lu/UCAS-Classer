# 打包准备

## 目标

这份文档只整理当前项目要实现“可打包、可安装、可独立运行”所需的工作。

当前代码以源码目录运行作为前提，适合本地开发验证，不适合直接打包分发。

## 当前不适合直接打包的原因

### 1. 路径依赖源码目录

当前 Rust 路径逻辑在 [paths.rs](/d:/lcy/ucasclasser-develop/src-tauri/src/paths.rs) 中基于 `CARGO_MANIFEST_DIR` 推导项目根目录。

这会带来两个问题：
- 打包后仍可能指向打包机器上的源码路径
- 数据文件、缓存文件、认证文件都默认落在仓库目录，不适合安装版

### 2. 运行时依赖 `npm run`

当前 Rust 调度层在 [script_runner.rs](/d:/lcy/ucasclasser-develop/src-tauri/src/script_runner.rs) 中直接调用：
- `npm run auth:check`
- `npm run auth:login`
- `npm run auth:reset`
- `npm run collect:all`

这意味着安装包如果要独立运行，必须解决这些依赖：
- `node`
- `npm`
- `tsx`
- `node_modules`
- TypeScript 源文件

当前安装版并不会自带这些东西。

### 3. Tauri 尚未声明运行时资源

当前 [tauri.conf.json](/d:/lcy/ucasclasser-develop/src-tauri/tauri.conf.json) 没有配置：
- `bundle.resources`
- `externalBin`

所以即使打包，也不会自动把后端运行所需脚本和依赖带进去。

### 4. 前端分发目录仍是占位状态

当前 `frontendDist` 指向 `../src`，但当前 `src/` 不是一个正式构建产物目录。

因此打包前还需要明确：
- 是继续做一个最小前端壳
- 还是先只做后端验证版

### 5. 运行数据目录策略还没切换

安装版不能继续把这些内容写回仓库目录：
- `storage-state.json`
- `data/cache/*.json`
- `data/ucas-classer.sqlite`

正式安装版应区分：
- 应用资源目录
- 用户数据目录
- 用户缓存目录

## 打包工作拆分

建议按下面顺序做，不要并行乱改。

### 第一阶段：路径系统改造

目标：
- 让打包版不再依赖源码根目录

要做的事：
- Rust 侧区分：
  - 应用资源目录
  - 用户数据目录
  - 用户缓存目录
- Auth 存储文件迁移到用户数据目录
- SQLite 迁移到用户数据目录
- 采集缓存迁移到用户缓存目录
- 保证开发态和打包态都能工作

验收标准：
- 不打开仓库目录，也能运行安装版
- 运行后生成的数据都不再写回源码仓库

### 第二阶段：去掉对本机 `npm run` 的依赖

目标：
- 打包后应用不依赖用户机器自己装的 Node 环境

推荐路线：
- 将 `automation/auth` 和 `automation/collectors` 的 TypeScript 编译成 JS
- 随包携带固定运行时
- Rust 调用固定 sidecar，而不是 `npm run`

可选路线：
- 随包带 `node.exe`
- Rust 直接运行：
  - `node <compiled-script>.js`

不推荐路线：
- 继续依赖用户机器全局 `npm`
- 继续依赖用户机器现成的 `tsx`

验收标准：
- 安装包在没有全局 Node/npm 的 Windows 机器上仍能执行：
  - auth check
  - auth login
  - collect
  - db import

### 第三阶段：声明 Tauri 打包资源

目标：
- 打包产物里真正包含运行时需要的文件

要做的事：
- 在 `tauri.conf.json` 中声明：
  - 后端 JS 产物
  - sidecar 可执行文件
  - 其他必要静态资源
- 明确安装后这些资源位于哪里
- 明确 Rust 如何定位这些资源

验收标准：
- 安装目录内能看到受控的运行时资源
- 安装版启动后能正确找到这些资源

### 第四阶段：收口前端壳与 Tauri 打包配置

目标：
- 让 Tauri 真正具备可打包前端入口

要做的事：
- 明确 `frontendDist`
- 准备最小前端构建产物
- 打开 `bundle.active`
- 明确 MSI / NSIS 的产物策略

验收标准：
- `tauri build` 能稳定输出安装包
- 安装包能正常启动主窗口

### 第五阶段：安装版专项验证

目标：
- 验证“安装后真实能跑”

至少要验证这些：
- 首次启动
- 手动登录并生成 `storage-state.json`
- `auth:check`
- runtime 定时 check
- runtime 自动 collect
- collect 后自动 import
- SQLite 数据完整性
- 中断后重新登录
- 关闭应用再打开，数据仍在

## 建议的新根目录策略

后续打包实验不要在当前仓库根目录直接做。

建议做法：

1. 保留当前目录作为稳定开发基线  
   当前目录：`d:\\lcy\\ucasclasser-develop`

2. 打包实验时复制到新目录  
   示例：
   - `d:\\lcy\\ucasclasser-package-test`
   - `d:\\lcy\\ucasclasser-bundle-lab`

3. 只在新目录里做这些高风险修改
   - 路径迁移
   - sidecar 引入
   - Tauri bundle 配置
   - 资源复制脚本
   - 安装包测试

4. 当前稳定目录只接受已经验证通过的改动回迁

这样做的好处：
- 当前开发根目录随时可回退
- 打包实验失败不会污染现有可用链路
- 可以做多轮安装测试而不影响主线

## 建议的打包实验流程

### Step 1

复制当前仓库到新的实验目录。

### Step 2

在实验目录里优先做路径改造，不动 auth/collect 业务逻辑。

### Step 3

把 `npm run` 调用替换为固定 sidecar 调用。

### Step 4

补齐 Tauri 打包资源配置。

### Step 5

生成安装包并做完整安装验证。

### Step 6

只有当安装版完整通过，再把稳定改动回迁到主目录。

## 当前已知最小打包完成标准

只要以下 6 件事都成立，就可以认为“安装版后端闭环成立”：

1. 安装版能启动
2. 安装版能登录并保存认证状态
3. 安装版能执行 auth check
4. 安装版能执行全量采集
5. 安装版能自动导入 SQLite
6. 安装版不依赖源码目录和本机全局 npm

## 当前结论

当前仓库适合继续做功能开发和本地验证。  
打包工作应在新的实验根目录中进行，并且优先解决：

1. 路径系统
2. Node/runtime sidecar
3. Tauri 资源打包
4. 安装版验证
